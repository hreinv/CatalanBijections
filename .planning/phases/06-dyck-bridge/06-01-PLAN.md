<task type="auto">
  <name>Task 1: Create bridge.js with bijection graph, BFS pathfinding, and step composition</name>
  <files>js/bijections/bridge.js</files>
  <action>
Create `js/bijections/bridge.js` implementing three capabilities:

**1. Bijection Graph Construction (`buildGraph`):**
- Import `structures` from `../structures/registry.js` to get all 11 structure keys.
- Build an adjacency list (Map) with all 11 structure keys as nodes.
- Add 8 classical edges (weight/priority: classical) from the known pairs. Hard-code the 8 classical edges from the existing bijection modules:
  - parentheses <-> dyck-path
  - binary-tree <-> triangulation
  - parentheses <-> binary-tree
  - ballot-sequence <-> dyck-path
  - dyck-path <-> mountain-range
  - dyck-path <-> lattice-path
  - binary-tree <-> rooted-plane-tree
  - non-crossing-partition <-> triangulation
- For "island" structures with zero classical edges (staircase-polygon, stack-sortable-perm), add bridge edges to 'dyck-path' as a hub node. These are type 'bridge'.
- Sort each node's neighbor list so classical edges come first (ensures BFS explores classical before bridge).
- Freeze the graph at module load time.

**2. BFS Shortest-Path (`findPath`):**
- Export `findPath(source, target)` returning `{ path: string[], edges: Array<{from, to, type}> }` or `null`.
- If source === target, return `{ path: [source], edges: [] }`.
- Standard BFS with path tracking. Because classical edges are sorted first in adjacency lists, BFS naturally prefers classical paths at equal depth.
- Should never return null when bridge edges are included (every structure is reachable).

**3. Step Composition (`composeSteps`):**
- Export `composeSteps(path, edges, dyckWord, n, classicalGetSteps)` where `classicalGetSteps` is the existing router's classical lookup function (passed as callback to avoid circular imports).
- For each leg of the path:
  - If edge type is 'classical': call `classicalGetSteps(from, to, dyckWord, n)` and spread into result array.
  - If edge type is 'bridge': create 2 simple bridge transition steps:
    - Step 1: description = `Converting ${sourceLabel} to Dyck word`, drawFrame renders source structure statically with full correspondence colors in the source box and an empty target box.
    - Step 2: description = `Building ${targetLabel} from Dyck word`, drawFrame renders target structure statically with full correspondence colors in the target box and an empty source box.
  - Bridge step drawFrame functions must import the structure modules from registry to call `fromDyck()` and `draw()` with the bounding boxes and colors from opts.
- Between multi-leg boundaries (NOT between bridge steps within same leg), insert a single transition step showing the intermediate structure rendered on both panels with description `Intermediate: ${intermediateLabel}`.
- Return the flat array of step objects (same contract as classical getSteps: `{ description, drawFrame }`).

**Key constraints:**
- Do NOT modify any existing classical bijection modules.
- Do NOT modify the animation engine.
- Bridge step drawFrame must accept the same `(ctx, easedProgress, opts)` signature as classical steps.
- Bridge drawFrame receives `opts.sourceBox`, `opts.targetBox`, `opts.theme`, `opts.colors` -- use these for rendering.
- For bridge steps, use the structure's standard `draw()` with correspondence colors (the `colors` param all draw() functions accept).
