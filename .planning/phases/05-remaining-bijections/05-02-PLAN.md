- getSteps(dyckWord, n, reversed) produces 2n+2 steps:
  - Step 0: Show Dyck path on grid (left panel), empty lattice n x n grid with dashed diagonal (right panel)
  - Steps 1..2n: For each step i, highlight Dyck segment i (up or down) and draw corresponding lattice segment (R or U). Dyck step +1 (up) maps to R (right), Dyck step -1 (down) maps to U (up). Description: "Step {i+1}: {up/down} path step maps to {Right/Up} lattice step"
  - Step 2n+1: Both paths fully colored with correspondence
- Drawing approach:
  - Left panel (Dyck): Copy drawDyckGrid/drawDyckSegments/drawDyckSegmentsComplete from parens-dyck.js pattern (self-contained helpers)
  - Right panel (Lattice): Draw n x n grid with dashed diagonal (y=x line). For coordinate transform, use lattice-path.js conventions: toCanvasX(gx) = offsetX + gx * cellSize, toCanvasY(gy) = offsetY + (n - gy) * cellSize (inverts y for canvas). Draw path segments with three-zone coloring. The lattice path builds incrementally: each R step goes right (+1,0), each U step goes up (0,+1).
  - CRITICAL (Pitfall 3 from RESEARCH.md): Use y-inversion `toCanvasY(gy) = offsetY + (n - gy) * cellSize` so U steps visually go upward on canvas. Copy coordinate transform from lattice-path.js draw() function.
  - Color: segment i in both panels gets `colors[i % colors.length]`
- Reversed: standard srcBox/tgtBox swap + steps.reverse()

**binary-plane-tree.js (BIJC-05):**
- META: source='binary-tree', target='rooted-plane-tree', label='Binary Trees to Rooted Plane Trees'
- Import binary-tree.js and rooted-plane-tree.js structure modules
- getSteps(dyckWord, n, reversed) produces n+2 steps:
  - Step 0: Show binary tree (left panel), plane tree root node only (right panel)
  - Steps 1..n: For each binary tree node i (pre-order), highlight the node in the binary tree and reveal the corresponding node in the plane tree. Description: "Node {i}: left child becomes first child, right child becomes next sibling"
  - Step n+1: Both trees fully colored with node correspondence
- Correspondence computation:
  - Build both trees from the Dyck word using fromDyck() on both structure modules
  - Walk the Dyck word to establish node-to-node correspondence: binary tree node i (0-indexed, pre-order) corresponds to plane tree node i+1 (pre-order, since plane tree root at index 0 has no binary tree counterpart). Per RESEARCH.md Pitfall 1: the plane tree has n+1 nodes (root + n children/descendants), binary tree has n nodes.
  - Collect binary tree nodes via pre-order traversal, collect plane tree nodes via pre-order traversal (excluding root as index 0). binaryNodes[i] corresponds to planeNodes[i+1].
- Drawing approach:
  - Left panel (Binary tree): Compute local layout using in-order traversal for x, depth for y (same pattern as binary-triang.js with _layoutX/_layoutY). Draw nodes as circles with edges. Three-zone coloring on nodes.
  - Right panel (Plane tree): Compute local layout using width-accumulation (same algorithm as rooted-plane-tree.js draw()). Draw nodes as circles with edges. Three-zone coloring. The root node is always drawn at default color (no binary tree counterpart). Child nodes colored by correspondence index.
  - Color: binary node i and plane tree node i+1 share `colors[i % colors.length]`
  - Pulse glow on active node pair, dimming on pending nodes at 0.25 alpha
- CRITICAL: Pre-compute all layouts in getSteps closure (not inside drawFrame). drawFrame only reads pre-computed positions.
- Reversed: standard srcBox/tgtBox swap + steps.reverse()

Both modules should be approximately 250-400 lines.
